01 · Objetivo

O que este projeto se propõe a resolver

  • Ligar um sistema de pedidos a um gateway de pagamento externo sem acoplar a regra de negócio aos detalhes da API do gateway.
  • Expor endpoints REST documentados, prontos para serem consumidos por qualquer cliente HTTP — não só a aplicação de teste.
  • Tratar o ciclo de vida de uma cobrança (pendente → aprovada, recusada ou cancelada) de forma rastreável e consultável.
  • Usar DTOs como contrato explícito de entrada e saída, e tratamento de exceções como parte do design da API, não como reação a bugs.

03 · Fluxo de integração

Do pedido ao status final

  1. Cadastrar pedido
    POST /api/pedidos — descrição e valor.
  2. Gerar pagamento
    POST /api/pagamentos cria uma preferência no Mercado Pago (POST /checkout/preferences), grava o pagamento como pendente e devolve a URL de checkout.
  3. Pagador completa o checkout
    Fluxo acontece na página do Mercado Pago, ambiente sandbox.
  4. Consultar status
    GET /api/pagamentos/{id}/status busca em GET /v1/payments/search pelo external_reference do pedido e atualiza o status local para aprovado, recusado ou cancelado.
  5. Histórico
    GET /api/pagamentos — lista tudo o que já foi gerado, direto da base local.

06 · Dificuldades e aprendizados

O que exigiu mais decisão

  • Chamar a API REST do Mercado Pago diretamente com RestClient em vez do SDK oficial — evita depender da assinatura exata de métodos de uma versão específica de biblioteca.
  • Mapear os sete status possíveis do gateway (approved, pending, in_process, rejected, cancelled, refunded, charged_back) para um vocabulário interno de quatro estados.
  • Ordenar os resultados de /v1/payments/search por data de criação, para sempre considerar a tentativa de pagamento mais recente de cada pedido.
  • Separar o tratamento de exceções por natureza do erro: validação (400), regra de negócio como pedido já pago (409) e falha de comunicação com o gateway (502).
